## v0.2.0 — First public build (Forge 1.20.1 server-side incremental backup)

BetterBackup is the backup branch of the Shinoyuki optimization series: BetterAutoSave (BAS) is for save/load IO, BetterBackup is for world backups. It is server-side only and integrates deeply with BAS — every time BAS finishes a chunk save it notifies BetterBackup, which backs up only the chunks that just changed, entirely on background worker threads with zero main-thread cost. The series follows one priority order: data safety > stability > compatibility > performance — any technique that would trade data fidelity, server stability, or compatibility for a smaller footprint or more speed is off the table.

It targets the two pain points of existing 1.20.1 backup mods: each backup zips the whole world, so 84 backups equal 84x the world size (a 200 GB world piles up past 16 TB in a year); and the main thread freezes for tens of seconds to minutes when a backup starts, kicking players. BetterBackup backs up only changed chunks and runs the whole thing on worker threads, cutting the footprint of 84 backups from the ~16.8 TB a vanilla approach needs down to roughly 150-300 GB, with 0ms of main-thread time per backup.

### What this build does

- Content-hash-deduplicated pack storage: it hashes the raw chunk-slot bytes inside the region file directly (XXH128 by default), so semantically identical bytes are stored once across snapshots, sidestepping the byte-instability of vanilla NBT serialization. Content-addressed objects are appended sequentially into large append-only pack files (immutable once sealed at a default 256 MiB), backed by an off-heap mmap sorted index, curing the random-seek cost of a million small files on a spinning disk; after a crash the index is rebuilt by scanning packs sequentially, a half-written tail record is truncated to the last complete boundary, and a stale index is never used
- Async incremental snapshots: capture follows BAS save events and processes only the chunks BAS just saved, never actively scanning the whole world; capture, hashing and writes all happen on worker threads the main thread never feels. INTERVAL mode defaults to one snapshot every 120 minutes
- Final snapshot on shutdown: on a normal server stop, after BAS has joined its own workers, a closing snapshot runs so every change before shutdown is captured
- Fine-grained restore:
  - Online single-chunk / area restore `restore-chunk-live`: applied live without stopping the server; a player with no args restores the chunk they stand in, `<radius>` 0-8 is a (2r+1)^2 area centered on the player; console / RCON use the explicit form `restore-chunk-live <id> <dim> <x> <z> [radius]`; above 9 chunks (radius>=2) it requires `confirm` within 30 seconds
  - Offline full / chunk-level CLI restore: the mod jar doubles as a command-line tool (bare JRE 17, no Minecraft / Forge needed), so you can self-rescue from a backup even when the server is down, with subcommands `list` / `info` / `verify` / `restore` / `fsck`; `restore` defaults to the full snapshot and also supports partial restore of terrain only via `--chunk <x>,<z>` for a single chunk or `--center <x>,<z> --radius <r>` for a rectangular area (everything else is left untouched)
  - In-game full restore `restore`: writes a flag file prompting a stop, then rebuilds before vanilla loads the world on the next startup, moving the current world's `region/` `entities/` `data/` `level.dat` `playerdata/` etc. to `<worldRoot>.bak-<timestamp>/` so the world and player inventory / advancements / statistics return to the same snapshot, avoiding the "world rolled back but inventory did not, letting players dupe" mismatch
- Rolling retention (off by default): `retention.enabled` defaults to false, every snapshot is kept forever and no history is ever pruned, so upgrading never starts deleting history implicitly; once enabled it prunes by the hourly / daily / weekly / monthly quotas. Three guards (the latest snapshot / the latest baseline-complete snapshot / the pending-restore target) are never pruned and also block manual mis-deletion; before enabling it, run `/betterbackup retention preview` for a dry-run of exactly which snapshots one prune pass would delete
- verifyOnSnapshot write-path self-check (off by default): after each snapshot it recomputes hashes for the objects newly written this snapshot; a mismatch marks only that snapshot `.incomplete` and does not trip degraded mode; it covers this snapshot's writes only, with a full-store integrity scan available via the CLI `fsck`. It is slow, and during the baseline full scan a single snapshot window can accumulate a large set of new objects making the re-check take minutes, so it defaults to off
- maxStoreSizeGB over-threshold guard (default 500): a soft, one-shot startup trigger, not a real-time disk quota and not a ring buffer. When startup detects the store above this value, a background pass first prunes expired manifests then runs a full GC to reclaim dead objects; reclaimable bytes are capped by the live data your retention policy still protects, so it WARNs when it cannot get under the value, and actually lowering the ceiling requires tightening retention
- Prometheus metrics (off by default): an opt-in, independent monitoring HTTP endpoint on port 9451 by default (avoiding BAS's 9450); on public servers firewall the port or bind it to 127.0.0.1

### Data safety and engineering quality

Before release it went through a full security audit and was hardened point by point, all around the series' top priority — data safety:

- Backups never touch world/: throughout a backup it only reads `.mca` files and writes into `backup-store/`; only restore ever writes the world
- Bad data never enters a backup: chunks read from disk are integrity-checked, and hitting half-written data mid-write is caught and retried, so no garbage bytes end up in a backup
- kill -9 tolerance: all writes use `tmp + fsync + atomic rename`, so a force-killed process never leaves a half-written file polluting the store, and leftover `.tmp` files are cleaned up on startup
- In-flight object protection during GC: a live-server manual GC is passed the in-flight hash protection set and does not seal the pack being written, so it cannot wrongly delete objects that are already stored and registered but not yet persisted with any manifest — which would otherwise be a silent data loss
- Recoverable mid-write pack failure: after an IO failure partway through appending to a pack, the rebuild preserves the `position == writeOffset` invariant so bad bytes do not corrupt the next append position
- Transactional restore: a full restore treats "move the existing world into `.bak`" as one transaction that rolls back on mid-way failure, so the world is never left half-new half-old
- A corrupt manifest never wrongly deletes the store: a full GC that hits an unreadable manifest aborts, never treating the objects it references as orphans
- BAS degradation is not silent: when a BAS background worker degrades on error, backups pause automatically and subsequent snapshots are marked incomplete; after a restart the changes within the degraded window are re-captured automatically
- 266 unit tests cover the store / snapshot / GC / torn read / baseline / player data / CLI / degraded re-capture paths; push and PR runs go through a CI gate that runs the root build including the full test suite

### Installation

- Minecraft 1.20.1 + Minecraft Forge 47.3.22+ + Java 17 (Eclipse Temurin)
- Server-side only (the dependency is declared side=SERVER)
- Requires [BetterAutoSave](https://github.com/ShinoyukiMiyako/Shinoyuki-BetterAutoSave/releases) >= 0.16.2 (this lower bound matches the dependency versionRange in mods.toml; below it Forge refuses to load BetterBackup). The online single-chunk restore additionally depends on the `SaveCoordination` / `ChunkRestoreOutcome` / `ChunkRestoreResult` API that only exists in BAS 0.16.2+
- Install the jar with the `-all` suffix (`shinoyuki_betterbackup-0.2.0-all.jar`, which bundles the hashing library); the one without `-all` lacks it and throws `ClassNotFoundException` on startup
- The first startup runs a rate-limited background scan of the entire existing world into the baseline (50 chunks/s by default); restore is refused until the baseline completes (otherwise the restored world would be missing a large area), and `/betterbackup status` shows the scan progress

### Configuration and commands

- Key defaults: `general.enabled=true`, `storage.hashAlgorithm=XXH128`, `storage.maxStoreSizeGB=500`, `schedule.mode=INTERVAL`, `schedule.intervalMinutes=120`, `retention.enabled=false`, `workers.backupWorkerThreads=2`, `safety.verifyOnStartup=true`, `safety.verifyOnSnapshot=false`, `prometheus.enabled=false`, `prometheus.port=9451`
- Config file `config/Shinoyuki-Optimize/shinoyuki_betterbackup/common.toml`; the full options, command table and restore flow are in the README
