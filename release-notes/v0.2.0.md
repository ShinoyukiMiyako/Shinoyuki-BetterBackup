## v0.2.0 — 首个公开版本（Forge 1.20.1 服务端增量备份）

BetterBackup 是 Shinoyuki 优化系列的备份分支：BetterAutoSave（BAS）之于存盘 / 加载 IO，BetterBackup 之于世界备份。服务端专用，与 BAS 深度集成——BAS 每完成一次 chunk save 就通知 BetterBackup，备份只处理刚变化的 chunk，全程跑在后台 worker 线程，主线程零开销。系列遵循同一条优先级：数据安全 > 稳定性 > 兼容性 > 性能——任何需要牺牲数据保真、服务器稳定或兼容性来省占用或提速的手段，一律不做。

它针对现有 1.20.1 备份 mod 的两个痛点：每次备份把整个世界打成一个 zip，84 份就是 84 倍世界大小（200 GB 世界一年累积超过 16 TB）；备份开始时主线程冻结几十秒到几分钟，玩家被踢。BetterBackup 只备份变化的 chunk 且整套在 worker 线程跑，把 84 份备份的占用从 vanilla 方案的约 16.8 TB 压到约 150-300 GB，单次备份主线程耗时 0ms。

### 本版能力

- 基于内容哈希去重的 pack 存储：直接对 region 文件里的 chunk slot 原始字节做哈希（默认 XXH128），语义相同的字节跨快照只存一份，绕开 vanilla NBT 序列化的字节不稳定问题。众多内容寻址对象顺序追加进 append-only 大 pack 文件（默认 256 MiB 封口后不可变），配 off-heap mmap 排序索引，根治机械盘上百万小文件的随机寻道；崩溃后顺序扫 pack 重建索引，写一半的尾部记录截断到上一条完整边界，绝不用过期索引
- 异步增量快照：跟随 BAS 的 save 事件采集，只处理 BAS 刚保存过的 chunk，从不主动扫描整个世界；采集、哈希、写入全在 worker 线程完成，主线程感知不到。INTERVAL 模式默认每 120 分钟一份
- 关服 final snapshot：服务端正常停止时，在 BAS join 完自己的 worker 之后跑一次收尾快照，确保关服前的所有变化都进备份
- 细粒度恢复：
  - 在线单 chunk / 区域回退 `restore-chunk-live`：不停服即时执行，玩家零参回退站位所在区块，`<radius>` 0-8 以站位为中心 (2r+1)² 块；控制台 / RCON 用显式形态 `restore-chunk-live <id> <dim> <x> <z> [radius]`；超过 9 块（radius≥2）需 30 秒内 `confirm` 二次确认
  - 离线全量 / 区块级 CLI 回退：mod jar 直接当命令行工具跑（裸 JRE 17，不需要 Minecraft / Forge），服务端瘫痪时照样能从备份自救，子命令 `list` / `info` / `verify` / `restore` / `fsck`；`restore` 默认整份全量回退，也支持 `--chunk <x>,<z>` 单块或 `--center <x>,<z> --radius <r>` 矩形区域的部分回退（只回地形，其余数据不动）
  - 游戏内全量恢复 `restore`：写 flag 文件提示停服，重启时在 vanilla 加载世界之前重建，把当前世界的 `region/` `entities/` `data/` `level.dat` `playerdata/` 等移到 `<worldRoot>.bak-<timestamp>/`，让世界与玩家背包 / 成就 / 统计回到同一份快照，不出现回档后背包与世界对不上的刷物品问题
- retention 滚动保留（默认关闭）：`retention.enabled` 默认 false，所有快照永久保留、不淘汰任何历史，升级首跑绝不隐式删历史；显式开启后才按 hourly / daily / weekly / monthly 配额滚动淘汰。三门禁（最新一份 / 最新 baseline 完整快照 / 待恢复目标）永不被淘汰，也拦手动误删；启用前先跑 `/betterbackup retention preview` dry-run 看清一次淘汰将删哪些快照
- verifyOnSnapshot 写入期自检（默认关闭）：每次快照后对本次新写入的对象重算哈希做写路径自检，失配只把该快照标记为 `.incomplete`，不触发降级模式；覆盖本次写入而非整个 store，全 store 完整性扫描走 CLI `fsck`。较慢，且 baseline 全量扫描期间单个快照窗口可能积累大量新对象令复验耗时数分钟，故默认关闭
- maxStoreSizeGB 超阈值守卫（默认 500）：软性、一次性的启动触发，非实时磁盘配额也非环形缓冲。启动检测 store 体积超过该值时，后台先 prune 掉超期 manifest 再跑一次全量 GC 回收死对象；可回收字节受 retention 仍保护的存活数据封顶，压不下去时给 WARN，真要压低上限需收紧 retention
- Prometheus 指标（默认关闭）：opt-in 的独立监控 HTTP 端口，默认 9451（避开 BAS 的 9450），公网服请用防火墙挡端口或改绑 127.0.0.1

### 数据安全与工程质量

发版前经过一轮全量安全审计并逐条加固，全部围绕系列首位优先级——数据安全：

- 备份不动 world/：整个备份过程只读取 `.mca` 文件、只写入 `backup-store/`，只有 restore 才会写世界
- 坏数据进不了备份：从磁盘读 chunk 时做完整性校验，撞上正在写入的半截数据会被拦下重试，不会把垃圾字节存进备份
- kill -9 容错：所有写入走 `tmp + fsync + atomic rename`，进程被强杀不会留半写文件污染 store，启动时自动清理 `.tmp` 残留
- GC 在途对象保护：活服手动 GC 传入在途 hash 保护集且不封口在写 pack，避免误删「已入库、已登记状态、尚未随任何 manifest 落盘」的在途对象造成静默丢数据
- pack 写入中途失败可恢复：pack 追加写中途 IO 失败后重建时保持 `position == writeOffset` 不变量，坏字节不会污染后续追加位置
- 恢复流程事务化：全量恢复把「现有世界搬进 `.bak`」做成整体事务，中途失败会 rollback，不会把世界拆散成半新半旧
- 损坏 manifest 不误删 store：全量 GC 遇到读不出的 manifest 直接 abort，绝不把它引用的对象当孤儿删掉
- BAS 降级不静默：BAS 后台 worker 异常降级时自动暂停备份、把后续快照标记为不完整，重启后自动补采降级窗口内的变化
- 266 个单元测试覆盖 store / 快照 / GC / 撕裂读 / baseline / 玩家数据 / CLI / 降级补采等路径；push 与 PR 走 CI 门禁，跑根 build 含全量测试

### 安装

- Minecraft 1.20.1 + Minecraft Forge 47.3.22+ + Java 17（Eclipse Temurin）
- 仅服务端安装（依赖声明 side=SERVER）
- 硬依赖 [BetterAutoSave](https://github.com/ShinoyukiMiyako/Shinoyuki-BetterAutoSave/releases) >= 0.16.2（版本下限与 mods.toml 的依赖 versionRange 一致，低于此版本 Forge 会拒绝加载 BetterBackup）。在线单 chunk 回退还依赖 BAS 0.16.2 才具备的 `SaveCoordination` / `ChunkRestoreOutcome` / `ChunkRestoreResult` API
- 装带 `-all` 后缀的 jar（`shinoyuki_betterbackup-0.2.0-all.jar`，内嵌哈希库）；不带 `-all` 的缺哈希库，启动报 `ClassNotFoundException`
- 首次启动会在后台限速扫描整个现有世界纳入基线（默认 50 chunk/s），基线完成前 restore 会被拒绝（否则恢复出的世界会缺一大块），`/betterbackup status` 可看扫描进度

### 配置与命令

- 关键默认值：`general.enabled=true`、`storage.hashAlgorithm=XXH128`、`storage.maxStoreSizeGB=500`、`schedule.mode=INTERVAL`、`schedule.intervalMinutes=120`、`retention.enabled=false`、`workers.backupWorkerThreads=2`、`safety.verifyOnStartup=true`、`safety.verifyOnSnapshot=false`、`prometheus.enabled=false`、`prometheus.port=9451`
- 配置文件 `config/Shinoyuki-Optimize/shinoyuki_betterbackup/common.toml`；完整配置项、命令表与恢复流程见 README
